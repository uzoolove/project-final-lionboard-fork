## 1. 요구사항 정의서

시스템이 제공할 기능과 품질 기준을 정리한 문서.  
구현 방법이 아닌 필요한 결과와 기대 수준을 기술함.

### 구성

- 문서 정보: 목적, 대상, 범위
- 시스템 개요: 해결 문제와 핵심 가치
- 사용자 유형: 역할 정의
- 기능/비기능 요구사항: 표 형식(ID, 우선순위, 내용)
- 제약 및 가정

### 작성 팁

- 사용자 관점 문장("사용자는 ~할 수 있어야 한다")
- 하나의 요구사항에 하나의 행동만
- 모호한 표현 대신 구체적 조건
- 우선순위 명시(1순위/2순위/3순위, 필수/권장/선택)

### 예시

| ID | 우선순위 | 요구사항 |
| --- | --- | --- |
| FR-01 | 1순위 | 사용자는 이메일과 비밀번호로 회원가입할 수 있어야 한다. |
| NFR-01 | 필수 | 로그인 정보는 안전하게 저장되어야 한다. |

## 2. 기능 정의서

요구사항을 실제 동작 관점으로 구체화한 문서.  
기능별 입력/처리/출력/예외를 정리함.

### 구성

- 공통 규칙: 권한, 에러 처리
- 기능 상세: 표 형식(기능ID, 기능명, 우선순위, 입력, 처리, 출력, 예외)

### 작성 팁

- 기능 단위로 분리하고 단계별 흐름 기술
- 예외 상황 최소 1개 이상 포함
- 권한 조건 명시(비회원/회원/작성자)

### 예시

| 기능ID | 기능명 | 우선순위 | 입력 | 처리 | 출력 | 예외 |
| --- | --- | --- | --- | --- | --- | --- |
| FD-01 | 로그인 | 1순위 | 이메일, 비밀번호 | 인증 성공 시 토큰 발급 | 로그인 성공, 게시판 이동 | 비밀번호 불일치 |

## 3. 화면 정의서

사용자 흐름과 화면 구성을 정리한 문서.  
화면별 목적, 구성 요소, 행동, 상태를 기술함.

### 구성

- 화면 목록: 표 형식(화면명, 목적, 주요 요소)
- 화면 흐름: 사용자 여정(다이어그램)
- 화면별 상세: 입력 항목, 행동, 상태 + 와이어프레임

### 작성 팁

- 한 화면당 하나의 명확한 목적
- 주요 상태(빈 화면, 오류, 권한 없음) 기록
- 화면 이동 경로 명시
- 와이어프레임은 구성 요소 배치 정도만

### 예시

| 화면명 | 목적 | 주요 요소 |
| --- | --- | --- |
| 게시판 목록 | 게시글 탐색 | 게시판 탭, 검색, 글 목록, 글쓰기 버튼 |

#### 게시판 목록 상세

- 구성: 게시판 탭, 검색 입력, 글 목록
- 행동: 탭 전환, 검색, 상세 이동
- 상태: 결과 없음 안내

## 4. 시스템 설계서

전체 구조와 데이터 흐름을 정리한 문서.  
기술 스택과 모듈 책임을 명확히 하여 구현 방향 공유.

### 구성

- 문서 목적
- 기술 스택
- 시스템 아키텍처
- 주요 구성 요소(라우팅, 상태 관리, 데이터 처리)
- 데이터 흐름
- 환경 변수/연동
- 보안 및 정책

### 작성 팁

- 구현 상세보다 구조와 책임 중심
- 실제 사용 기술만 기술
- 데이터 흐름은 단계별 순서로 간결하게

### 예시

#### 데이터 흐름

1. 서버 컴포넌트에서 데이터 조회
2. 폼 제출 시 Server Action에서 API 호출
3. 성공 시 캐시 갱신 후 페이지 이동
4. 로그인 시 클라이언트에서 사용자 정보 저장
